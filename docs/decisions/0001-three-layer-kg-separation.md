# 0001 — Three-layer KG separation (TBox / SKOS / ABox)

**Status:** accepted

## Context

UX Methods needs to evolve a thin coordinating taxonomy into a real knowledge graph without entangling stable semantics with content-derived facts. Content (methods, IO concepts) is authored in Sanity and changes frequently; the semantic model of "what a method is and how it relates to its inputs and outputs" changes much less often.

If curated semantics and exported content end up in the same file or load path, every content edit risks bleeding into the contract, and every contract change risks invalidating exported data without anyone noticing.

## Decision

The graph is split into three layers, each owned by a different mechanism:

1. **Ontology (TBox)** — `graph/ontologies/uxmethods-core.ttl`. Curated by hand, version-controlled, treated as a stable contract. Includes alignments such as PROV.
2. **Taxonomy (SKOS)** — `graph/build/io-taxonomy.ttl`. Generated by `graph/scripts/push-io-taxonomy.js` from Sanity. Disposable.
3. **Methods data (ABox)** — `graph/build/methods-data.ttl`. Generated by `graph/scripts/method-export.js` from Sanity. Disposable.

Both generators emit `owl:Ontology` headers so Protégé imports them by logical IRI rather than physical file path.

## Consequences

- **Regenerate, don't edit.** Hand-edits to files in `graph/build/` are not allowed; they will be overwritten on the next export. Behavioural changes to generated data are made by editing Sanity content or the exporter scripts.
- **Stable contract surface.** Downstream tools (eventual plan builder / method-stack generator) depend on stable IRIs and predicates from the ontology layer. Renames in the TBox are breaking changes.
- **Independent evolution.** Schema work in Sanity does not require ontology changes; ontology refactors do not require content republishing (beyond a re-export).
- **Local prototyping works.** `graph/ontologies/workspace.ttl` plus a Protégé catalog wraps all three layers into one merged model for SPARQL iteration.
