# 0007 — Content-model Mermaid plugin: in-monorepo first, extract later

**Status:** accepted

**Related:** [ADR 0006](0006-content-model-mermaid-export.md) — the content-model Mermaid export and the `probe` / `walker` / `emit-mermaid` contract this plugin reuses.

## Context

[ADR 0006](0006-content-model-mermaid-export.md) established a Mermaid `classDiagram` export of the Sanity content model, generated by the `content-model/` CLI. The CLI loads `studio/schemaTypes/index.ts` directly via `tsx`. That direct-import path has a known, documented limitation: **types contributed by Studio plugins are invisible to it.** `sanity-plugin-taxonomy-manager` adds `skosConcept` / `skosConceptScheme` at plugin-initialisation time, not in `schemaTypes/index.ts`, so the CLI never sees them — fields referencing `skosConcept` (e.g. `method.topics`, `method.input`, `method.output`) render their type label but their edges drop with warnings. This is visible in the committed `docs/content-model.md`: `SkosConcept` appears as a field type but there is no `SkosConcept` class and no edges to it.

The fix the CLI's architecture always anticipated: run the same pipeline **inside Studio**, reading the **fully-composed** workspace schema via `useSchema()` — which includes every plugin-contributed type. That turns the artefact into a Sanity Studio plugin that renders the diagram in-Studio, with room for filter and export UI.

The question this ADR settles is **where that plugin is developed and how it reaches a standalone form**, not the rendering contract (0006 owns that).

## Decision

**Develop the plugin in-monorepo first, as a new workspace shaped like a standalone plugin from the start; extract it to its own repository later, adding only the publishing/CI shell at that point.**

### Workspace

- New pnpm workspace `sanity-plugin-mermaid-content-model/`, sibling to `studio/`, `astro/`, `graph/`, `content-model/`. The directory name matches the eventual npm package and repository name so extraction is a 1:1 carve-out (`git filter-repo --subdirectory-filter`).
- Internals mirror the canonical `@sanity/plugin-kit` / `@sanity/pkg-utils` layout (modelled on [`bobinska-dev/sanity-plugin-rich-table`](https://github.com/bobinska-dev/sanity-plugin-rich-table), already a dependency of `studio/`):
  - `@sanity/pkg-utils` build (`package.config.ts`) → `dist/` + types.
  - `exports` map with a `source` condition (`./src/index.tsx`) and a `default` condition (`./dist/index.js`). In dev, the host studio resolves `source` and runs the plugin's **`src/` directly — no build step during development**.
  - Three-tsconfig split: `tsconfig.settings.json` (shared, strict — carries 0006's `noUncheckedIndexedAccess` / `exactOptionalPropertyTypes` / `verbatimModuleSyntax`), `tsconfig.json` (dev/typecheck: src + configs), `tsconfig.dist.json` (build: src minus tests).
  - Vitest for tests.

### The real studio is the test bed

While in-monorepo, the **production `studio/`** plays the role rich-table gives its bundled dev studio: it adds the plugin via `"sanity-plugin-mermaid-content-model": "workspace:*"` and registers it in `sanity.config.ts`. This is strictly better than a synthetic test studio for our purpose — `studio/` has `sanity-plugin-taxonomy-manager`, so `useSchema()` actually surfaces `skosConcept` and we can verify the orphaned-edge fix against live content.

### Pure modules are copied, not shared

`probe.ts`, `walker.ts`, and `emit-mermaid.ts` (and their test suites) are **copied** from `content-model/src/` into the plugin's `src/`. The `content-model/` CLI remains the reference implementation. A shared `content-model-core` workspace was considered and rejected: it would itself have to publish or extract for the standalone plugin to build, which defeats the low-drama-extraction goal. The [ADR 0006](0006-content-model-mermaid-export.md) contract plus duplicated test suites pin the two copies against drift. (The only mechanical change on copy: relative imports drop their explicit `.ts` extensions, since the plugin is bundled by pkg-utils rather than run under `tsx`.)

### Adopt now vs. adopt at extraction

| Adopt **now** (in-monorepo) | Adopt **at extraction** (standalone repo) |
| --- | --- |
| pkg-utils build + `source`/`default` exports | Built-in dev `studio/` (synthetic) |
| Three-tsconfig split | semantic-release + conventional commits + commitlint |
| Vitest | husky + lint-staged |
| `definePlugin` entry, copied pure modules | CI matrix + cross-version `test:studio-install` |
| | `plugin-kit link-watch`, npm publish, public package |

The right-hand column is all *outer shell* — release automation and a synthetic studio. None of it touches plugin code, so adding it later does not perturb `src/`, the exports map, or the build config. This is what makes the extraction low-drama, and it lets the plugin deliver working value before any publishing machinery exists (Gall's Law — see [docs/design-principles.md](../design-principles.md)).

## Consequences

- The plugin sits **parallel to** the three KG layers (like the content-model artefact in [ADR 0006](0006-content-model-mermaid-export.md)): it is not part of the ontology/taxonomy/methods graph, not pushed to Fuseki, not involved in inference.
- Two copies of `probe`/`walker`/`emit-mermaid` now exist (CLI + plugin). They must be kept behaviourally aligned; the shared test suites are the guard. If they need to diverge, that divergence should be deliberate and documented.
- The plugin must consume whatever shape `useSchema()` returns. This is expected to be the **compiled** `Schema` / `SchemaType` model, not the raw `defineType` objects the walker consumes — so a schema-adapter module is the first real build cycle, and it (not the walker) absorbs any shape difference. The walker stays unchanged.
- Studio target is **Sanity v5 / React 19** (the current state of `studio/`), not the v3 assumed in the original handoff.
- Extraction will require widening `private: true` to a public package, adding `publishConfig`, and flipping `studio/`'s `workspace:*` dependency to the published version.
